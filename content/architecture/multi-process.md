# Chromium 多进程架构

> 理解 Chrome 为什么是多进程的，以及每个进程的职责

---

## 为什么需要多进程？

早期浏览器是单进程的，一个标签页崩溃 = 整个浏览器崩溃。Chrome 从诞生之初就采用了多进程架构，核心目标：

1. **隔离性**：每个标签页独立进程，一个崩溃不影响其他
2. **安全性**：渲染进程沙盒化，即使被攻破也无法直接访问系统
3. **性能**：多进程可以利用多核 CPU

## 进程类型

```
┌─────────────────────────────────────────┐
│          Browser Process (1个)           │
│  - UI 线程                               │
│  - IO 线程                               │
│  - 网络请求                              │
│  - 扩展管理                              │
│  - 标签页管理                            │
└──────┬──────────┬──────────┬────────────┘
       │          │          │
┌──────▼──┐ ┌────▼────┐ ┌───▼────┐
│Renderer │ │Renderer │ │ GPU    │
│Process  │ │Process  │ │Process │
│(每Tab)  │ │(每Tab)  │ │(共享)  │
└─────────┘ └─────────┘ └────────┘
```

### Browser Process

浏览器主进程，负责：
- UI 渲染（地址栏、标签栏、菜单）
- 网络请求调度
- 扩展管理
- 标签页生命周期管理
- 文件系统访问

### Renderer Process

每个标签页（或站点实例）一个渲染进程，负责：
- HTML 解析
- CSS 计算
- JavaScript 执行（V8）
- DOM 树构建
- 布局计算

### GPU Process

所有标签页共享一个 GPU 进程，负责：
- GPU 加速渲染
- 合成（Compositing）
- 视频解码

### Utility Process

临时进程，用于：
- 文件解析（PDF、XML）
- 网络服务（DNS、代理）
- 扩展的 Service Worker

## Site Isolation

Chrome 的站点隔离确保不同域名的页面运行在不同的渲染进程中：

- **Process-per-Site**：同一域名共享进程
- **Process-per-Site-Instance**：同一域名的不同浏览上下文可能使用不同进程

这是 Spectre 漏洞防御的关键机制。

## IPC 机制

Chrome 使用 **Mojo** 作为跨进程通信框架：

```
Process A                    Process B
    │                            │
    │  ┌─────────────────────┐   │
    ├──►│  Mojo Message Pipe  ├──►│
    │  └─────────────────────┘   │
    │                            │
```

Mojo 的优势：
- 类型安全（接口定义在 .mojom 文件中）
- 异步优先
- 支持零拷贝传输
- 自动生成多语言绑定

---

## 进程模型决策

Chrome 通过 `SiteInstance` 决定页面应该运行在哪个进程：

1. 新导航到达
2. 计算目标 URL 的 `SiteInstance`
3. 如果已有匹配的进程，复用
4. 否则创建新进程
5. 如果进程数超过上限（默认约 40-50），复用最不活跃的进程
